## API access — reality (web-verified, Jun 2026)

No end-user "log in and pull my data" API at most U.S. banks. U.S. open banking runs on the **FDX** standard,
**CFPB-recognized as the standard-setting body (Jan 2025)**; **114M+ consumer accounts** connected via FDX
APIs (Q1 2025, +50% YoY), but access is for **authorized data recipients via consent/OAuth**, brokered by
aggregators — not direct keys for an end user. CFPB §1033 rollout ongoing.
Legend: ✅ public dev API (usable for the relevant data) · 🔑 consumer-data API via FDX/OAuth for **authorized
recipients** (not self-serve) · ⚙️ business/treasury or payments-partner only (not consumer txn data) ·
🔌 aggregator-only · ❌ none.

## Direct pull paths that don't need an aggregator

| Path | What | Why it matters for import |
|---|---|---|

| **Brokerage APIs** | Schwab/IBKR/E*TRADE/Tradier/Alpaca direct | direct holdings/txns with user's own key |
| **Crypto exchange APIs** | Coinbase/Kraken/Gemini/etc. | direct holdings/txns with user's own key |

## Implications for CashFlux (local-first)

| Option | Fit |
|---|---|
| File import (CSV/XLSX/OFX/PDF) | ✅ primary — no backend, fully local (C74) |
| **OFX Direct Connect** lane | ⚠️ possible direct pull (user creds), no aggregator; brittle/declining |
| Brokerage/crypto APIs (BYO key) | ⚠️ optional direct lane for holdings/txns; needs opt-in |
| Aggregators (Plaid/MX/Akoya/SnapTrade) | ❌ needs a hosted backend + stored consent — against local-first SPEC |
